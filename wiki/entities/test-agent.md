# Test Agent (TST)

A specialized agent in [[autodata|AutoData's]] [[development-squad|Development Squad]] responsible for debugging, creating comprehensive test cases, and verifying that generated code functions correctly before production execution.

## Role and Responsibilities

The Test Agent serves as AutoData's quality assurance specialist, ensuring that code generated by the [[engineering-agent|Engineering Agent]] is robust, handles edge cases, and produces expected outputs. TST acts as the critical checkpoint between code generation and production validation, catching bugs early in the development cycle.

Key responsibilities include:
- **Test case design**: Creating comprehensive unit and integration tests
- **Debugging**: Identifying and diagnosing code errors
- **Edge case coverage**: Testing pagination, missing data, malformed responses
- **Execution verification**: Running code in controlled environments
- **Error reporting**: Providing detailed feedback to [[engineering-agent|ENG]] for fixes
- **Regression testing**: Ensuring fixes don't break existing functionality
- **Performance testing**: Checking execution time and resource usage

For example, TST might test a stock price scraper with: valid ticker symbols, invalid tickers, API rate limiting scenarios, network timeouts, malformed JSON responses, and missing data fields.

## Architecture Integration

TST operates between the [[engineering-agent|Engineering Agent]] and [[validation-agent|Validation Agent]] in the Development Squad workflow. It receives code from ENG, runs tests in a sandboxed environment, and either approves the code for VAL execution or sends detailed error reports back to ENG for refinement.

This iterative loop continues until code passes all tests:
```
ENG → Code → TST → Tests Pass → VAL
              ↑       ↓
              └── Tests Fail ←┘
```

The Test Agent prevents the [[validation-agent|VAL]] from executing flawed code on real websites, which could waste API quotas, trigger anti-scraping defenses, or produce incorrect data.

## Tools and Capabilities

The Test Agent employs multiple testing frameworks and techniques:
- **Unit testing**: pytest, unittest for function-level verification
- **Mock environments**: Creating fake responses to test parsing logic
- **Sandbox execution**: Running code in isolated environments
- **Code analysis**: Static analysis to identify potential bugs
- **Coverage tracking**: Ensuring all code paths are tested
- **Performance profiling**: Measuring execution time and memory usage

Following the [[react-paradigm|ReAct paradigm]], TST operates iteratively:
1. **Reasoning**: Analyzes code to identify potential failure points
2. **Acting**: Creates and executes test cases
3. **Observation**: Reviews test results and error messages
4. **Reporting**: Provides actionable feedback to ENG or approval to proceed

## Interaction with Other Agents

**Direct collaboration**:
- Receives code from [[engineering-agent|ENG]]
- Provides test results and bug reports back to ENG
- Approves code for [[validation-agent|VAL]] when tests pass
- Coordinates with [[manager-agent|MGR]] on iteration limits

**Upstream dependencies**:
- Test scenarios informed by [[plan-agent|PLN's]] task decomposition
- Edge cases based on [[web-agent|WEB's]] structural analysis
- API test cases based on [[tool-agent|TOL's]] documentation findings
- Success criteria from [[blueprint-agent|BLU's]] specifications

The Test Agent's thoroughness directly affects the reliability of AutoData's final output.

## Communication via OHCache

TST benefits from [[ohcache|OHCache's]] efficient communication:
- **Code access**: Retrieves generated code from [[local-cache-system|Local Cache System]]
- **Test result storage**: Caches test cases for similar tasks
- **Selective reporting**: Only sends failure details to ENG, not full test logs
- **Structured feedback**: Uses [[oriented-hyperedge-formatter|Oriented Hyperedge Formatter]] for bug reports

This targeted communication reduces token costs while ensuring ENG receives actionable debugging information.

## Test Case Categories

TST designs diverse test scenarios:

**Functional tests**:
- Correct data extraction from valid inputs
- Proper handling of expected variations
- Data format compliance with specifications

**Error handling tests**:
- Network failures and timeouts
- HTTP error codes (404, 500, 429)
- Malformed HTML or JSON responses
- Missing or unexpected data fields

**Edge case tests**:
- Empty result sets
- Pagination boundaries
- Rate limiting and throttling
- Authentication failures

**Performance tests**:
- Execution time within acceptable limits
- Memory usage for large datasets
- Concurrent request handling

## Debugging Capabilities

When tests fail, TST provides detailed diagnostics:
- **Error location**: Specific line numbers and functions
- **Error type**: Exception class and message
- **Context**: Input data that triggered the failure
- **Expected vs actual**: What should have happened vs what did
- **Suggested fixes**: Potential solutions based on error pattern

This precision enables [[engineering-agent|ENG]] to quickly resolve issues without guessing.

## Performance Impact

The Test Agent's quality assurance is critical for AutoData's success:
- **Catch bugs early**: Prevents wasted validation attempts on broken code
- **Reduce iteration cycles**: Clear debugging info minimizes ENG rework
- **Improve robustness**: Comprehensive edge case testing ensures reliability

In the [[instruct2ds|Instruct2DS]] benchmark, TST's thorough testing contributed to:
- **High completion rate**: Most tasks succeed on first validation attempt
- **Cost efficiency**: $0.57 average per task (minimal wasted validation runs)
- **Speed**: 5.58 minutes average includes TST's quick test cycles

Without TST, AutoData would experience higher failure rates and cost overruns from executing untested code.

## Iterative Refinement

TST enables rapid improvement cycles:
1. **First iteration**: ENG generates initial code → TST finds basic errors
2. **Second iteration**: ENG fixes bugs → TST finds edge case issues
3. **Third iteration**: ENG adds error handling → TST approves for validation

This typically converges within 2-3 iterations, demonstrating efficient collaboration between ENG and TST.
